# Video-Factory Contract

**Version:** 1.0
**Status:** Stable
**Last Updated:** 2026-03-09

## Overview

Tree Identity generates video manifests as JSON files in Cloudflare R2. The video-factory service consumes these manifests to produce videos. This document defines the interface contract between the two systems.

## Manifest Location

```
s3://{R2_BUCKET}/manifests/{slug}.json
```

- **Bucket:** Configured via `R2_BUCKET` env var
- **Key pattern:** `manifests/{slug}.json` where `slug` is the content piece's URL slug
- **Format:** JSON, UTF-8 encoded

### HTTP Access

Manifests are also available via HTTP API:

```
GET /api/manifests/{slug}
```

Returns `200` with JSON body, `404` if not found, `400` if slug is invalid.
Cache: `public, s-maxage=60, stale-while-revalidate=300`.

## Manifest Schema

```json
{
  "slug": "my-article",
  "title": "My Article Title",
  "type": "articles",
  "style": "cinematic",
  "sections": [
    {
      "id": "intro",
      "timestamp": "00:00",
      "narration": "Welcome to this article about...",
      "bRollQuery": "technology workspace",
      "onScreenText": "Introduction",
      "mediaRefs": ["media-ref-1", "media-ref-2"]
    }
  ],
  "cover": {
    "url": "https://cdn.example.com/cover.jpg",
    "alt": "Article cover image"
  },
  "publishedAt": "2026-03-09T10:00:00.000Z",
  "treeIdentityVersion": "1.0"
}
```

### Field Reference

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `slug` | string | Yes | URL slug, unique identifier |
| `title` | string | Yes | Content title |
| `type` | string | Yes | Collection slug: `articles`, `notes`, or `records` |
| `style` | string \| null | No | Video style: `cinematic`, `tutorial`, or `vlog` |
| `sections` | array | Yes | Ordered list of video sections |
| `sections[].id` | string | Yes | Section identifier |
| `sections[].timestamp` | string \| null | No | Target timestamp (e.g. `"00:30"`) |
| `sections[].narration` | string \| null | No | Narration text for this section |
| `sections[].bRollQuery` | string \| null | No | Search query for B-roll footage |
| `sections[].onScreenText` | string \| null | No | Text overlay content |
| `sections[].mediaRefs` | string[] | No | References to media assets |
| `cover` | object \| null | No | Cover image metadata |
| `cover.url` | string | Yes | Image URL |
| `cover.alt` | string | Yes | Alt text |
| `publishedAt` | string | Yes | ISO 8601 publish timestamp |
| `treeIdentityVersion` | string | Yes | Contract version (currently `"1.0"`) |

## Trigger

Manifests are generated by the `generateVideoManifest` Payload afterChange hook:

- **Fires when:** `doc.status === 'published' && doc.video.enabled === true`
- **Collections:** Articles, Notes, Records
- **Operation:** Overwrites `manifests/{slug}.json` on every qualifying save

## Consumer Contract

- video-factory reads R2 directly via S3 API — no Payload API calls required
- Alternatively, use the HTTP API at `/api/manifests/{slug}`
- Consumers should poll or use event-driven triggers (e.g. R2 event notifications)
- Manifests are complete and self-contained — no additional lookups needed

## Versioning Policy

- The `treeIdentityVersion` field tracks the contract version
- **Minor changes** (new optional fields): backward-compatible, no version bump
- **Breaking changes** (field removal, type change): bump major version (e.g. `"2.0"`)
- Consumers should check `treeIdentityVersion` and handle unknown versions gracefully

## Error Handling

- Manifest upload failures are **non-blocking** — the content save succeeds even if R2 upload fails
- Errors are logged to server console: `[video-manifest] Failed to upload manifest for {slug}`
- video-factory should handle missing manifests gracefully (content may exist without a manifest)
- Stale manifests may exist if R2 upload failed on a subsequent update — consumers should check `publishedAt`
